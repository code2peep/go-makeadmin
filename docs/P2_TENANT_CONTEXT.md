# P2.2 Tenant Context

## 目标

P2.2 建立后台请求的租户上下文规则：

- 认证通过后，所有 makeadmin 运行链路都从 request context 读取 `tenant_id`。
- 默认租户仍为 `0`，表示全局默认上下文。
- 不修改数据库 schema。
- 不开放租户切换入口。
- 不连接或迁移 zyai 业务库。

## 租户来源

登录阶段：

- 未携带 `X-Tenant-ID` 时，使用默认租户 `0`。
- 携带 `X-Tenant-ID: 0` 时，仍使用默认租户 `0`。
- 携带非 `0` 的 `X-Tenant-ID` 时，P2.2 直接拒绝；等后续补租户成员校验后再开放。

认证后请求：

- 可信来源是 JWT claim `tenantId`。
- 请求可以携带 `X-Tenant-ID`，但必须和 JWT `tenantId` 一致。
- 未携带 `X-Tenant-ID` 时，直接使用 JWT `tenantId`。

当前 P2.1 登录只会签发 `tenantId=0` 的 JWT，因此 P2.2 的实际运行租户仍是全局默认上下文。

## 越权处理

认证后请求如果出现以下情况，直接返回无权限：

- `X-Tenant-ID` 不是无符号整数。
- `X-Tenant-ID` 与 JWT `tenantId` 不一致。

这能阻止客户端通过伪造请求头切换租户。

## 上下文写入

认证中间件通过后会写入：

- `context.Context`：供 adapter、service 和后续数据权限逻辑读取。
- `gin.Context`：保留当前请求内的便捷读取能力。

当前已从上下文读取 `tenant_id` 的运行链路：

- 管理员、角色、组织、岗位。
- 设置、存储、网站、协议、版权。
- 文件分类和文件元数据。
- 登录日志和操作日志。
- 首页配置和控制台信息。

菜单、权限定义和字典仍按全局能力表处理。

## 不在 P2.2 做

- 不新增租户切换 UI。
- 不新增租户成员校验表或迁移。
- 不做数据权限查询约束。
- 不做跨租户管理员授权。
- 不修改前端 header 名称。

## 后续任务

- P2.3：数据权限查询约束。
- P2.4：租户成员和租户切换入口。
- P2.5：租户级设置、文件和日志的迁移策略。
